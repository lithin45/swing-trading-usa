# Phase 1 Audit — swing-signals (2026-06-10)

**Scope:** full read-only audit of the repo at `main` (116b747) against the project mandate:
signal-only, %-based sizing, per-signal transparency, and the prime directive of **≤ 6–7 new
entries per calendar month as a ceiling, never a quota**. No code was changed.

**Method:** four parallel deep-read audits (data layer, signal core, backtest/validation,
broker/ops/security), each independently verified on the highest-stakes claims (budget
enforcement, exits mode, validation numbers, membership CSV). Full test suite run: **306 passed,
3 deselected** (~68s).

---

## 1. Executive summary

The system is far more built than the original brief assumes. Since the brief was written it has
gone through: a momentum-core redesign (V1), a deep review with same-day remediation of all
findings, broad-universe point-in-time backtesting, a pre-registered experiment program with an
untouched holdout, and a live Alpaca **paper**-trading + Streamlit dashboard layer (stage 8).
Engineering quality is high: clean module boundaries, 306 green tests, honest commit-message
science, fail-loud data handling, and a security posture that passed a hostile review.

**One finding dominates everything else:**

> **P0 — The prime directive is not enforced anywhere in code.** There is no monthly entry
> counter, no budget policy (default / time-decayed / ranked-queue), no per-name cooldown, no
> "budget exhausted" rejection class, and the realized trades-per-month **distribution** has never
> been measured. The system currently relies on selectivity depth alone (green-only regime gate,
> composite ≥ 70, agreement ≥ 0.70, extension veto, liquidity floor, momentum eligibility) to
> stay under 7/month. A genuinely hot month can and will exceed the ceiling — nothing rejects
> entry #8.

Verdict on the rest: the **selectivity machinery the budget needs already exists and is good**;
what's missing is the budget mechanism itself (mandate §4), the cadence proof (mandate §7), and
three specific validation upgrades (overfitting controls, loss-halt replay, persistent
artifacts). No subsystem needs a ground-up rebuild.

**Validation status (honest numbers):** the deployed config (legacy exits + no-chase 3.0 ATR +
3-ATR stop) validated at **+0.118R/trade, PF 1.27, win 54.7%, CAGR +6.6%** on the untouched
2025-01 → 2026-06 holdout, after scoring ≈ flat (−0.012R, PF 0.97) on the brutal 2022–24
in-sample window. The earlier **+0.239R / PF 1.48** figure (still in my session notes and some
prior reporting) was the curated 10-name watchlist re-validation and was later shown to be
**flattered by survivorship of the curated list** — the broad-universe numbers are the real
ones. The edge is real-but-thin and regime-dependent: it earns in trending tape and bleeds in
crash/chop (the combo's value-add is cutting 2022–23 bleed from −35.7R to −2.8R). With the
brief's expected 20–50% live degradation, the margin over zero is small. **Paper-first is the
correct current posture; no real capital yet.**

## 3. P0 — the missing trade-budget mechanism (mandate §4–5, §7)

What the mandate requires vs. what exists:

| Required | Exists today? |
|---|---|
| Monthly new-entry counter, persisted, calendar-month scoped | **No** — nothing counts entries per month anywhere (engine, persistence, broker gates) |
| Default policy: high absolute bar + hard cap | Half: the high bar exists (composite ≥ 70, agreement ≥ 0.70, green-only regime, extension veto); the cap does not |
| Optional variants (time-decayed bar / ranked queue) as backtestable config | **No** |
| Max concurrent open positions | ✅ Yes — `max_positions: 8` enforced per-run in the engine and pre-trade in `broker/gates.py` |
| Per-name cooldown after a stop | **No** — a stopped-out name can be re-signaled the next day |
| Log every rejected high-scoring setup with reason incl. "budget exhausted" | Partial — gate/threshold rejections are logged and persisted; the budget-exhausted class can't exist yet |
| Cadence check: realized trades/month **distribution**, confirming ~0–7 organically | **Never measured** — only a flat average is reported |

Notes on severity:
- `monthly_loss_halt: 0.10` (`broker/gates.py`) is a **loss**-based halt, not an entry-count cap —
  the only "monthly" control in the system, and it lives in the paper-broker layer, not the
  signal engine. Telegram alerts (what you actually trade from) have **no** monthly governor.
- The signal engine is stateless across days by design: it caps positions/heat/sector **per
  run**, then persistence dedupes per `(signal_date, symbol)`. Two strong signal days in one
  month → 8+ entries with zero pushback, and the dashboard would not even flag it.
- The current `max_positions: 8` is itself in tension with the spirit of the ceiling: 8
  concurrent slots with ~2–4 week holds arithmetically supports >7 entries/month.

This is the core Phase 2/3 design-and-build item. It is cleanly insertable: the engine already
produces a ranked, scored, explained candidate list; the budget is a thin stateful gate between
ranking and emission, with its counter in the existing SQLite/Postgres store and its policy in
config.

## 4. Validation status (mandate §7) — honest but incomplete

**What is genuinely good** (and unusual for a project at this stage):
- Backtest shares the **live** `generate_signals()` code path — no reimplementation drift.
- Point-in-time discipline: S&P 500 membership reconstructed per bar (`members_asof`), historical
  VIX/VIX3M from FRED, indicator panels sliced `index <= asof`, chandelier trail `shift(1)`
  (no same-bar lookahead; regression-tested).
- Realistic frictions: 10 bps/side, limit-in-zone fills with aging + market fallback, cash
  accounting (no implicit leverage), cross-day position/heat caps, liquidation costs in the
  final equity point.
- Pre-registered experiments with one mutation each, then an untouched holdout; negative results
  reported plainly (commit a20fea2).

**The numbers** (broad point-in-time S&P 500 universe, costs/caps/limit fills):

| Window | Config | Result |
|---|---|---|
| 2022–24 (in-sample, incl. momentum crash) | production base | −0.134R, PF 0.77 |
| 2022–24 | best combo (legacy exits + no-chase 3.0 + 3-ATR stop) | −0.012R, PF 0.97 |
| **2025-01 → 2026-06-09 (untouched holdout)** | **combo (deployed)** | **+0.118R, PF 1.27, win 54.7%, CAGR +6.6%** |
| same holdout | base | +0.125R, PF 1.24, CAGR +16.1%, maxDD −8.6% |
| 2017–2019 (second untouched window)* | combo | +0.112R, PF 1.28, CAGR +7.7%, maxDD −14.7% |
| 2018 alone (SPY −6.1%)* | combo vs old config | −0.102R / −7.4% vs −0.283R / −16.5% |

\* The 2017–2019 run (session of commit 2a7307d, after the cache fixes) exists **only in session
notes — it was never persisted to the repo**, which is exactly gap #4 below. Caveat recorded
with it: older windows carry a larger residual survivorship gap (87/580 union-membership names
unfetchable for 2017–19). Taken together, the combo has **two independent untouched windows
agreeing (~+0.11R, PF ~1.27 across ~445 OOS trades)**, including one bear year where it cut
losses ~2–3× vs the old config.

The legacy "+0.239R / PF 1.48" (commit 1130d3d, 10-name watchlist) is superseded — the curated
list itself embedded survivorship. Repo config comments already cite the honest numbers.

**Gaps against the mandate (all P1):**
1. **No overfitting controls.** No Deflated Sharpe, no PBO, no purged/combinatorial CV, no ±10%
   parameter-sensitivity sweeps — all explicitly required by `research/11_backtesting.md`.
   `walk_forward.py` is (self-admittedly) a regime-stability check, not an overfitting test, and
   there is no ledger of trials from which DSR could even be computed.
2. **Account-level halts not replayed in the backtest.** Daily/weekly/monthly loss halts and
   drawdown derisk/halt exist only in `broker/gates.py` (live). Backtested returns and cadence
   may be overstated relative to live behavior.
3. **Cadence distribution unreported** (required by mandate §7) — fold into the P0 work: the
   budget mechanism must be added to the backtester and the per-month histogram reported, showing
   the bar (not the cap) does the work.
4. **No persistent validation artifacts.** Experiment results live in `/tmp`, commit messages,
   and session notes — the 2017–2019 second-window evidence above is not in the repo at all.
   A go/no-go report per the mandate (metrics + caveats + degradation expectation) should be a
   versioned file in the repo.
5. **Regime coverage is real but thin and under-documented.** Two untouched OOS windows now
   agree, including the 2018 bear year — good — but the strategy remains regime-dependent by
   design (green-only momentum): profitable in trending tape (2017/2019/2024/2025–26),
   flat-to-losing in crash/chop (2018/2022/2023) with the combo cutting those losses ~2–3×. The
   go/no-go report should state this regime-conditional record explicitly rather than a single
   blended number.

**Current verdict: NO-GO for real capital; GO for continued paper forward-testing** (which is
exactly what is running). This matches the brief's own sequencing.

## 6. Keep / build / remove

**Keep as-is (sound):**
- Data layer post-fixes: providers + retry + normalize + union-merge cache + quality gate (`data/`)
- Factor stack and weights: f08 momentum core (45%), f01 technical (25%), f02 Claude news (20%),
  f09 setup (10%); all causal, all self-disabling on missing data
- Regime gate as true veto (`green_only_entries: true`, VIX hard veto, SPY 200-DMA) and macro
  size modifier
- Exit state machine (`exits.py`) with legacy mode live; staged mode kept as a researched,
  tested, validated-worse alternative behind config
- Risk/sizing: vol-scaled, conviction-tiered, heat/sector/notional/gross caps, loss halts
- Persistence + audit trail (runs/signals/trades/outcomes/snapshots), idempotent on re-runs
- Alerting fan-out + dead-man's switch; AI layer (memoized, graceful degradation)
- Broker paper layer (gates, entries, manage, orphan sweep) — impressively defensive
- Backtest runner core (shares live signal code; realistic fills/costs)

**Build (the Phase 2/3 work, in mandate order):**
1. Trade-budget module: persisted calendar-month counter + policy interface (default high-bar +
   hard cap; variants A/B as config) + per-name cooldown + budget-aware rejection logging —
   enforced in the **signal engine** (so Telegram alerts respect it), mirrored in the backtester
2. Cadence report: trades-per-month histogram in backtest report + dashboard
3. Validation upgrades: trial ledger + Deflated Sharpe + PBO (CSCV), ±10% parameter sweeps,
   loss-halt replay in the runner, persistent versioned validation reports in `docs/validation/`
4. Bar-finality guard on signal runs (refuse/warn on partial intraday bars)
5. Small ops hardening: intraday workflow concurrency group, orphan-adoption stop fallback
   (synthesize 2-ATR stop from last cached bars or escalate to ERROR alert), Neon preflight check

**Remove:** nothing. (f03 events / f05 themes / f06 smart-money were never built — config
documents this honestly; whether to build f03 for earnings-date gating should be decided in the
Phase 2 plan, since post-earnings drift was a mandated candidate archetype and an earnings-
calendar veto is cheap insurance for a multi-day holder.)

## 8. Top risks, ranked

1. **Ceiling breach in a hot month** — the prime directive currently depends on market weather,
   not code (P0 #1).
2. **Thin, regime-dependent edge + missing overfitting controls** — ~+0.11R / PF ~1.27 across
   two untouched OOS windows, but ≈ flat across 2022–24; after the brief's expected 20–50% live
   degradation the margin approaches zero. Mitigation is already in place (paper-only) — keep it
   that way until the validation gaps close and paper results accumulate.
3. **Backtest/live behavioral drift** — halts (and, once built, the budget) must be simulated,
   or reported cadence/returns overstate live.
4. **Silent data degradation** — partial-bar manual runs, stale-cache splits, silent provider
   dropout (P1 #2, #6, P2 #12).
5. **Ops single points** — Neon dependency, orphan stops, intraday race (P1 #7–9).
